# 接收代码评审

## 概述

代码评审需要技术评估，而非情绪表演。

**核心原则：** 先验证再实现。先询问再假设。技术正确性优先于社交舒适感。

## 响应模式

```
收到代码评审反馈时：

1. 阅读：完整读完反馈，不急于反应
2. 理解：用自己的话复述需求（或者提问）
3. 验证：对照代码库现状检查
4. 评估：对当前代码库而言技术上合理吗？
5. 回应：技术性确认或有理有据的反驳
6. 实现：逐项实现，每项都测试
```

## 禁止的回应

**绝不要：**

- "你说得完全对！"（明确违反 CLAUDE.md）
- "好观点！" / "很棒的反馈！"（表演性的）
- "让我现在就实现"（在验证之前）

**正确做法：**

- 复述技术需求
- 提出澄清问题
- 如果有误，用技术理由反驳
- 直接开始干活（行动 > 言辞）

## 处理不明确的反馈

```
如果有任何项目不明确：
  停止 - 先不要实现任何东西
  询问 不明确的项目

原因：各项可能相互关联。部分理解 = 错误的实现。
```

**示例：**

```
用户："修复 1-6"
你理解 1、2、3、6。对 4、5 不确定。

❌ 错误：先实现 1、2、3、6，之后再问 4、5
✅ 正确："我理解第 1、2、3、6 项。第 4 和第 5 项需要澄清后才能继续。"
```

## 按来源分类处理

### 来自用户

- **信任** — 理解后实现
- 如果范围不清楚**仍然要问**
- **不要表演性附和**
- **直接行动**或技术性确认

### 来自外部评审者

```
实现之前：
  1. 检查：对当前代码库而言技术上正确吗？
  2. 检查：是否会破坏现有功能？
  3. 检查：当前实现方式有何原因？
  4. 检查：是否在所有平台/版本上可行？
  5. 检查：评审者是否理解完整上下文？

如果建议看起来有误：
  用技术理由反驳

如果无法轻松验证：
  如实说明："没有 [X] 我无法验证这一点。我应该 [调查/询问/继续]？"

如果与用户之前的决策冲突：
  先停下来与用户讨论
```

**用户的规则：**"外部反馈——保持怀疑，但仔细核查"

## YAGNI 检查："专业"功能

```
如果评审者建议"正确实现"：
  grep 代码库检查实际使用情况

  如果未使用："这个端点没有被调用。移除它（YAGNI）？"
  如果在使用：那就正确实现
```

**用户的规则：**"你和评审者都向我汇报。如果我们不需要这个功能，就不要添加。"

## 实现顺序

```
对于多项反馈：
  1. 先澄清所有不明确的项
  2. 然后按以下顺序实现：
     - 阻塞性问题（崩溃、安全）
     - 简单修复（拼写、导入）
     - 复杂修复（重构、逻辑）
  3. 逐个测试每个修复
  4. 验证无回归
```

## 何时反驳

在以下情况反驳：

- 建议会破坏现有功能
- 评审者缺乏完整上下文
- 违反 YAGNI（未使用的功能）
- 对当前技术栈而言技术上不正确
- 存在遗留/兼容性原因
- 与用户的架构决策冲突

**如何反驳：**

- 使用技术推理，而非防御心态
- 提出具体问题
- 引用可工作的测试/代码
- 如果涉及架构，让用户参与

**如果不太方便公开反驳的暗号：**"Strange things are afoot at the Circle K"

## 确认正确的反馈

当反馈确实正确时：

```
✅ "已修复。[简要描述改了什么]"
✅ "好的发现 - [具体问题]。已在 [位置] 修复。"
✅ [直接修复并在代码中展示]

❌ "你说得完全对！"
❌ "好观点！"
❌ "感谢你发现了这个！"
❌ "感谢你 [任何事]"
❌ 任何感谢表达
```

**为什么不感谢：** 行动自己会说话。直接修复就好。代码本身就说明你接收到了反馈。

**如果发现自己快要写"感谢"：** 删掉它。改为描述修复内容。

## 优雅地纠正你的反驳

如果你反驳了但你是错的：

```
✅ "你是对的——我检查了 [X]，它确实 [Y]。正在实现。"
✅ "验证了这一点，你是对的。我最初的理解有误，因为 [原因]。正在修复。"

❌ 长篇道歉
❌ 辩护为什么你之前反驳了
❌ 过度解释
```

陈述事实性更正，然后继续。

## 常见错误

| 错误             | 修正                |
| ---------------- | ------------------- |
| 表演性附和       | 陈述需求或直接行动  |
| 盲目实现         | 先对照代码库验证    |
| 批量处理不测试   | 逐项实现，逐项测试  |
| 假设评审者是对的 | 检查是否会破坏功能  |
| 回避反驳         | 技术正确性 > 舒适感 |
| 部分实现         | 先澄清所有项目      |
| 无法验证仍然继续 | 说明限制，请求指导  |

## 真实示例

**表演性附和（错误）：**

```
评审者："移除遗留代码"
❌ "你说得完全对！让我移除那些..."
```

**技术验证（正确）：**

```
评审者："移除遗留代码"
✅ "检查中... 构建目标是 10.15+，这个 API 需要 13+。向后兼容需要保留遗留代码。当前实现 bundle ID 有误——是修复它还是放弃 13 以下支持？"
```

**YAGNI（正确）：**

```
评审者："实现完善的指标追踪，包括数据库、日期筛选、CSV 导出"
✅ "grep 了代码库——没有地方调用这个端点。移除它（YAGNI）？还是有我遗漏的使用场景？"
```

**不明确的项目（正确）：**

```
用户："修复第 1-6 项"
你理解 1、2、3、6。对 4、5 不确定。
✅ "理解第 1、2、3、6 项。第 4 和第 5 项需要澄清后才能实现。"
```

## GitHub 评论回复

在 GitHub 上回复行内评审评论时，在评论线程中回复（`gh api repos/{owner}/{repo}/pulls/{pr}/comments/{id}/replies`），而非发表顶层 PR 评论。

## 底线

**外部反馈 = 待评估的建议，而非待执行的命令。**

验证。质疑。然后实现。

不要表演性附和。始终保持技术严谨。
