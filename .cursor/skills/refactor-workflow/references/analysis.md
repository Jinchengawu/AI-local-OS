# 重构分析方法

## 分析流程

```
代码异味检测 → 标准符合性检查 → 影响范围评估 → 输出报告
```

## 1. 代码异味检测

### 通用异味

| 异味     | 阈值    | 检测方法      |
| -------- | ------- | ------------- |
| 大文件   | >400 行 | `wc -l 文件`  |
| 大函数   | >50 行  | 读取代码分析  |
| 深嵌套   | >4 层   | 读取代码分析  |
| 重复代码 | -       | 对比相似文件  |
| 魔术数字 | -       | grep 数字常量 |

### 前端异味

参见 `coding-standards` skill 的"代码异味检测"章节。

## 2. 标准符合性检查

**首先必须加载编码规范**：

- 调用 `coding-standards` skill，或读取 `.cursor/skills/coding-standards/` 下所有规范文件
- 必须读取：`references/api.md`、`references/react.md`、`references/typescript.md`

然后对照规范逐条检查：

- 命名规范
- 类型安全
- 错误处理
- 不可变性
- 组件职责分离
- 提取 Hook 信号
- 避免的坏味道
- **API 请求规范**（禁止 ahooks useRequest，必须用 @tanstack/react-query）

## 3. 影响范围评估

### 依赖分析

```bash
# 谁依赖这个文件
grep -rn "from.*目标文件名" src/

# 这个文件依赖谁
grep -n "^import" 目标文件

# 统计依赖数量
grep -rn "from.*目标文件名" src/ | wc -l
```

### 风险评估

| 影响范围 | 依赖文件数 | 风险等级 |
| -------- | ---------- | -------- |
| 局部     | 0-2        | LOW      |
| 中等     | 3-10       | MEDIUM   |
| 广泛     | >10        | HIGH     |

### 高风险场景

- 被多处导入的公共组件/工具
- 没有测试覆盖的代码
- 涉及状态管理的逻辑
- 与 API 交互的代码
- 涉及交易/风控/资金的代码

## 4. 分析报告模板

```markdown
## 重构分析报告

### 目标

- 文件：[路径]
- 类型：[前端/后端/通用]
- 行数：[X 行]

### 代码异味

| 异味     | 当前值 | 阈值 | 状态  |
| -------- | ------ | ---- | ----- |
| 文件行数 | X      | 400  | ✅/❌ |
| ...      |        |      |       |

### 标准符合性

**coding-standards** (X/Y 通过)：

- ✅ [通过的规则]
- ❌ [违反的规则] - [具体问题]

### 影响范围

- 依赖此文件：[X] 个文件
- 风险等级：[HIGH/MEDIUM/LOW]

### 重构建议

1. [建议1] - 解决 [问题]
2. [建议2] - 解决 [问题]
```
