## 评审清单

**代码质量：**

- 关注点分离是否清晰？
- 错误处理是否恰当？
- 类型安全（如适用）？
- 是否遵循 DRY 原则？
- 是否处理了边界情况？

**架构：**

- 设计决策是否合理？
- 是否考虑了可扩展性？
- 是否有性能影响？
- 是否有安全隐患？

**测试：**

- 测试是否真正测试了逻辑（而非 mock）？
- 是否覆盖了边界情况？
- 是否在需要的地方有集成测试？
- 所有测试是否通过？

**需求：**

- 是否满足所有计划需求？
- 实现是否匹配规格？
- 是否有范围蔓延？
- 是否记录了破坏性变更？

**生产就绪性：**

- 迁移策略（如有 schema 变更）？
- 是否考虑了向后兼容？
- 文档是否完整？
- 是否有明显 bug？

## 关键规则

**要做的：**

- 按实际严重程度分类（不是所有都是严重）
- 要具体（文件:行号，不要含糊）
- 解释问题为什么重要
- 承认优点
- 给出明确判定

**不要做的：**

- 没检查就说"看起来不错"
- 把小问题标为严重
- 对没有评审过的代码给反馈
- 含糊其辞（"改进错误处理"）
- 回避给出明确判定
